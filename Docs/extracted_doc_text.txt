3.3 M-Pesa integration notes
The Daraja API (STK Push) is the standard method for collecting M-Pesa payments in Kenya. You need a Safaricom business account and a registered shortcode or Paybill number.

Register at developer.safaricom.co.ke and get sandbox credentials first
Use the lipa-na-mpesa-online (STK Push) endpoint
The flow: user clicks Pay, enters phone number, M-Pesa prompt appears on their phone, they enter PIN, server receives callback confirming payment
Store every transaction in the payments table regardless of status
Handle callbacks asynchronously â€” do not rely on redirect for payment confirmation
Go live requires a Safaricom business account and KYC documents

5. Security
5.1 Authentication security
Passwords hashed using bcrypt (minimum 12 rounds) â€” handled automatically by Supabase Auth
JWT tokens with 1-hour expiry, refresh tokens with 7-day sliding window
Rate limiting on login endpoint: max 5 failed attempts before 15-minute lockout
Email verification required before full account access
Google OAuth uses official Supabase OAuth flow â€” no password stored for OAuth users

5.2 Payment security
Never store card details â€” all card processing via Flutterwave (PCI DSS compliant)
M-Pesa callbacks validated using Safaricom IP whitelist and shared secret
All payment amounts validated server-side â€” never trust client-sent amounts
Duplicate payment protection: check payment reference before granting access
Payments table is append-only â€” no record deletion

5.3 Data security
All data in transit encrypted via HTTPS (TLS 1.2 minimum, 1.3 preferred)
Supabase database at rest encryption enabled
Row-level security (RLS) enabled on all Supabase tables â€” users can only read their own data
Admin routes protected by role check on every request
Environment variables for all secrets â€” never hardcoded in source

5.4 Content security
YouTube embeds use privacy-enhanced mode (youtube-nocookie.com)
Content Security Policy (CSP) headers configured on all pages
Admin panel access restricted by IP whitelist in production (optional but recommended)
Input sanitisation on all user-facing text fields to prevent XSS

5.5 Operational security
Supabase backups enabled (daily automated backups)
Error logging via Sentry â€” no sensitive data in logs
Dependency audit: run npm audit weekly, patch critical vulnerabilities within 48 hours
Two-factor authentication on all admin accounts
